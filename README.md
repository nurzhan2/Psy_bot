# Telegram-бот по психологии (RAG + GPT)

Бот-ассистент для проекта Екатерины: ведёт терапевтический диалог по заданному
алгоритму, опираясь на базу знаний из личных материалов и теоретической литературы.

## Структура проекта

```
psy-bot/
├── app/
│   ├── handlers/
│   │   ├── commands.py     # /start, /reset
│   │   ├── onboarding.py   # дисклеймер, согласия, проверка 18+
│   │   └── dialogue.py     # основной диалог с пользователем
│   ├── rag/                 # загрузка документов, эмбеддинги, поиск (FAISS)
│   ├── db/
│   │   ├── history.py      # история диалогов
│   │   └── user_state.py   # согласия, возраст, антифлуд-счётчик
│   ├── prompts.py          # системный промпт, табу, лимиты, антифлуд
│   ├── legal_texts.py      # тексты дисклеймера и согласий
│   ├── safety.py           # проверка на кризисные сообщения
│   └── llm.py               # сборка запроса к OpenAI
├── tools/                   # вспомогательные скрипты (см. ниже)
├── books/                   # конспекты литературы (см. README.txt внутри)
├── personal_materials/      # личные материалы Екатерины (уже частично заполнено)
├── data/                    # БД и векторные индексы (создаются автоматически)
├── config.py
├── main.py
├── requirements.txt
├── .env                     # уже заполнен реальными ключами — см. предупреждение ниже
└── .env.example
```

## ⚠️ Важно про .env и ключи доступа

`.env` с реальными ключами **больше не лежит в архиве** — он был засвечен
в переписке и в git-истории (GitHub Push Protection заблокировал пуш именно
по этой причине). См. **`SECURITY_FIX.md`** — там пошаговая инструкция:
1. Перевыпустить ключ OpenAI и токен бота (обязательно)
2. Убрать `.env` из git-истории
3. Создать `.env` заново из `.env.example` с новыми ключами

Старые ключи, которые были в переписке, считай скомпрометированными —
использовать их дальше нельзя, даже если технически они ещё активны.

- `private_channels_posts.txt` — 184 поста из приватных каналов с практиками

## Онбординг (юридический блок)

При первом `/start` бот:
1. Показывает дисклеймер об ответственности
2. Предлагает три РАЗДЕЛЬНЫЕ галочки (нельзя объединять по 152-ФЗ, ст. 9):
   - оферта + пользовательское соглашение (обязательно)
   - обработка персональных данных (обязательно)
   - рекламная рассылка (необязательно)
3. Просит подтвердить возраст 18+
4. Только после этого открывается доступ к диалогу

Тексты — в `app/legal_texts.py`, составлены на основе документов от Екатерины.
Это рабочий черновик, финальные формулировки должна подтвердить она/юрист.

## Лимиты и защита

- Сообщение длиннее 4000 символов отклоняется с пояснением (`MAX_MESSAGE_LENGTH`)
- Таймаут ответа от OpenAI — 30 секунд, иначе бот пишет, что сервер перегружен
- После ~12 сообщений без чёткого запроса бот мягко возвращает к формулировке задачи
- История хранится по Telegram `user_id`, не по имени — это и есть обезличивание

## Подключение Telegram-каналов (уже сделано, для справки на будущее)

Если появятся новые материалы из каналов — используй `tools/parse_channel_export.py`.
Скрипт работает с ПОЛНЫМ экспортом аккаунта (Settings → Advanced → Export Telegram
Data, без выбора одного диалога — получится файл со всеми чатами сразу).

Посмотреть, какие чаты есть в файле:
```bash
python tools/parse_channel_export.py path/to/result.json --list
```

Выгрузить нужные чаты по точному названию (можно несколько сразу):
```bash
python tools/parse_channel_export.py path/to/result.json personal_materials/новый_файл.txt \
    --include "Точное название чата 1" "Точное название чата 2"
```

После выгрузки — анализ стиля (отдельно от контента, без копирования текста):
```bash
python tools/analyze_style.py personal_materials/public_channel_posts.txt
```
Результат — текстовое описание стиля, которое нужно руками вставить в
`SYSTEM_PROMPT` в `app/prompts.py`.

Не забудь пересобрать индекс после любых изменений в `personal_materials/`:
```bash
python -m app.rag.build_index
```

## Голосовые сообщения и аудио

**От пользователей** (`app/handlers/voice.py`): бот принимает голосовые, расшифровывает через Whisper API, и СРАЗУ удаляет временный аудиофайл (в `finally`, так что удаление происходит даже при ошибке распознавания) — это прямое юридическое требование клиентки (п.2, запрет на хранение биометрии). Дальше распознанный текст идёт по той же логике, что и обычное сообщение (онбординг/лимиты/кризис-фильтр/антифлуд).

**Авторские медитации Екатерины** (`tools/transcribe_audio.py`): отдельный скрипт для расшифровки ЕЁ собственных аудио-практик в текст для базы знаний — это не персональные данные клиента, поэтому ограничение на хранение не действует. Нужно прогнать вручную (требует доступа к OpenAI API):
```bash
python tools/transcribe_audio.py personal_materials/audio_raw/
python -m app.rag.build_index
```

## Что уже реализовано

- Диалог через aiogram (3.x) с сохранением истории в SQLite
- RAG с разделением на 2 приоритета источников: personal (выше) и books (ниже)
- Системный промпт на основе детального ТЗ клиентки: роль, тон, алгоритм из 4 шагов,
  и 8 жёстких сценариев табу (диагнозы/лекарства, суицид, гадания, директивность,
  защита базы знаний, 18+, нейтралитет к политике/религии, запрет на выдумки)
- Кризисный фильтр (суицид/самоповреждение) — срабатывает раньше похода к GPT,
  с ссылкой на личную консультацию Екатерины и на экстренные службы
- Онбординг с дисклеймером и тремя раздельными согласиями + проверка 18+
- Лимит длины сообщения, таймаут ответа, антифлуд
- Обезличивание данных (хранение по `user_id`)
- Приём и обработка голосовых сообщений с немедленным удалением аудио после распознавания
- Бот вежливо отклоняет неподдерживаемые типы сообщений (фото/видео/документы)
- Личные материалы Екатерины (публичный + приватные каналы, практики) уже в проекте
